# AgentProof 랜딩페이지 다음 버전 개선 계획

- 문서 버전: 1.0
- 기준 버전: V4.1
- 목표 버전: V5 계열
- 원칙: 기능을 먼저 늘리지 않고 고객 행동 데이터로 다음 버전을 결정한다.

---

## 1. 개선 원칙

V5는 “더 화려한 랜딩페이지”가 아니라, V4.1에서 확인된 고객의 문제와 전환 행동을 바탕으로 **자가진단과 결과 전달을 제품화하는 버전**이다.

다음 기능은 근거 없이 선행 개발하지 않는다.

- 복잡한 로그인
- 고객별 SaaS 대시보드
- 자동 규제 적합성 보증
- 결제·구독
- 수십 개 AI 도구 연동
- 대규모 관리자 기능

- 무료 자료와 유료 파일럿 사이의 이탈 이유

---

## 3. V5 판단 시점

다음 중 늦은 시점을 기준으로 V5 범위를 확정한다.

- V4.1 공개 후 4주
- 유효 세션 300건 이상
- 정상 리드 제출 20건 이상

표본이 부족하면 개발 대신 유입과 메시지 실험을 연장한다.

---

## 4. 핵심 성공 지표

아래 수치는 업계 평균이 아니라 내부 의사결정 기준이다.

| 지표 | 관찰 기준 | 의미 |
|---|---:|---|
| 진단 화면 클릭률 | 20% 이상 | 결과물 관심 |
| 모달 오픈률 | 8% 이상 | 문제·오퍼 적합성 |
| 폼 시작/모달 오픈 | 55% 이상 | 폼 첫 인상 적합 |
| 제출/폼 시작 | 35% 이상 | 폼 마찰 허용 범위 |
| 제출/세션 | 2.5% 이상 | 전체 전환 가능성 |
| 적합 리드/제출 | 30% 이상 | 타깃 정확성 |
| 상담 예약/적합 리드 | 25% 이상 | 후속 가치 |
| 파일럿 관심/상담 | 20% 이상 | 지불 가능성 신호 |

수치가 낮다고 즉시 기능을 추가하지 않는다. 어느 단계에서 이탈하는지 먼저 확인한다.

---

## 5. V5 제품 가설

> 방문자가 8~12개의 짧은 질문에 답하면, 역할과 업무 상황에 맞춘 AI 준비도 결과를 즉시 보고 이메일로 요약 리포트를 받을 수 있다면, 단순 문의 폼보다 더 많은 문제 데이터와 적합 리드를 확보할 수 있다.

V5의 핵심은 “자동 진단처럼 보이는 설문”이 아니라, 사용자가 자신의 상황을 구조화하고 다음 행동을 이해하게 만드는 것이다.

## 6. 버전별 로드맵

## V4.1 — Production Hardening

목표: 현재 랜딩페이지를 실제 유입과 리드 수집에 사용할 수 있게 한다.

- 실제 폼 저장
- UTM·분석
- 개인정보처리방침
- 반응형·접근성
- 안정적인 배포

완료 조건: QA 문서 통과.

## V4.2 — 메시지 실험

목표: 어떤 역할과 문제 메시지가 가장 강한지 확인한다.

- 히어로 카피 A/B
- CTA 카피 A/B
- 역할별 캠페인 URL
- 폼 필드 순서 실험
- 샘플 화면 클릭 위치 분석

복잡한 기능 개발 없이 콘텐츠와 전환 구조만 실험한다.

## V5.0 — Self-serve AI Readiness Check

목표: 3분 자가진단과 개인화 결과 제공.

- 8~12문항 분기형 진단
- 역할별 질문 흐름
- 업무 유형과 위험 요인 점수화
- 결과 페이지
- 이메일 결과 링크
- 상담 CTA

## V5.1 — Report Automation

목표: 샘플이 아닌 실제 응답 기반 요약 리포트 생성.

- 1페이지 HTML/PDF 리포트
- 위험 지도
- 우선 도입 업무
- 정책 공백
- 다음 액션
- 운영자 검토 후 발송 옵션

## V5.2 — Lead Operations

목표: 고객검증 데이터를 반복 분석할 수 있게 한다.

- 최소 관리자 화면
- 리드 상태 관리
- 역할·문제·유입별 필터
- 후속 연락 기록
- CSV 내보내기
- 삭제 요청 처리

## 7. V5.0 진단 설계 초안

### 단계 1: 역할

- 실무자
- 대표·임원·팀장
- 도입 담당자
- 보안·정책·감사 담당자
- 구축·컨설팅 담당자

### 단계 2: 사용 단계

- 아직 사용하지 않음
- 개인적으로 사용 중
- 일부 팀 사용 중
- 파일럿 준비
- 전사 도입 검토
- 정책 수립 중

### 단계 3: 업무 유형

- 문서·보고서 작성
- 조사·요약
- 고객응대
- 사내 지식 검색
- 코드·데이터 분석
- 외부 발송·자동화

### 단계 4: 데이터 민감도

- 공개 정보
- 일반 사내자료
- 고객·계약 정보
- 개인정보
- 재무·법무·인사 정보

### 단계 5: 검토와 책임

- 사람 검토 없음
- 개인 판단
- 팀장 승인
- 보안·법무 검토
- 로그·승인 체계 있음

### 결과 축

- 도입 기회
- 정확성 위험
- 정보보호 위험
- 책임·승인 공백
- 실행 준비도

점수는 의사결정 보조이며 법적 적합성을 보증하지 않는다고 명시한다.

## 8. 결과 페이지 구성

1. 한 줄 상태: 예) `도입 가능 · 기준 보완`
2. 역할별 요약
3. 우선 적용 업무 1~3개
4. 조건부 적용 업무
5. 제한·금지 검토 업무
6. 가장 큰 정책 공백 3개
7. 30일 실행 체크리스트
8. 이메일 리포트 받기
9. 파일럿 상담 CTA

결과를 보기 위해 이메일을 먼저 강제하지 않는 실험과, 이메일 이후 전체 결과를 제공하는 실험을 비교한다.

## 9. 실험 백로그

| ID | 가설 | 변경 | 핵심 지표 | 성공 판단 |
|---|---|---|---|---|
| EX-01 | 보편적 가치 문구가 더 많은 적합 리드를 만든다 | 현재 H1 vs `직원들은 이미 AI를 씁니다` | qualified lead/session | 15% 이상 개선 |
| EX-02 | 결과 중심 CTA가 문의형 CTA보다 강하다 | `AI 준비도 확인` vs `샘플 결과 보기` | modal open | 15% 이상 개선 |
| EX-03 | 폼을 2단계로 나누면 완료율이 높다 | 단일 폼 vs 역할/문제 후 연락처 | submit/form start | 15% 이상 개선 |
| EX-04 | 역할별 진입 페이지가 적합도를 높인다 | 공통 랜딩 vs persona query | qualified/submit | 10% 이상 개선 |
| EX-05 | 샘플 리포트 공개가 신뢰를 높인다 | 목업만 vs 1페이지 샘플 | submit/session | 10% 이상 개선 |
| EX-06 | 초기 단계 고지가 전환을 해치지 않는다 | 보조 문구 위치 변경 | modal open, submit | 비열등 + 신뢰 피드백 |

한 번에 하나의 핵심 변수만 변경한다.

## 11. 기술 변화 계획

V5.0에서 추가될 가능성이 높은 구조:

```text
app/
├─ check/page.tsx
├─ check/result/[token]/page.tsx
├─ api/assessments/route.ts
├─ api/reports/route.ts
components/assessment/
lib/scoring/
lib/reporting/
```

추가 데이터 모델:

- assessments
- assessment_answers
- assessment_results
- report_deliveries

원칙:

- 점수 규칙은 코드와 문서에 버전 관리한다.
- 결과 token은 예측 불가능해야 한다.
- 개인정보와 답변을 분리 저장할 수 있게 설계한다.
- 동일 응답에 같은 버전의 점수 규칙을 적용하면 동일 결과가 나와야 한다.

## 12. 실제 신뢰 요소 추가 조건

다음은 증거가 생긴 뒤에만 추가한다.

### 고객 로고

- 서면 공개 동의 확보.

### 사례 수치

- 측정 방법과 기간 확인.
- 예: 시간 절감, 위험 발견 수, 정책 수립 완료 등.

### 후기

- 실제 담당자의 승인된 문구.
- 직함·회사 공개 범위 확인.

### 보안·규제 표현

- 외부 프레임워크 참고와 인증을 구분.
- `준수 보장`, `인증 완료` 같은 표현은 실제 근거 없이는 금지.

## 13. V5 진입 의사결정

### 경우 A — 방문은 많고 모달 오픈이 낮음

- 기능 개발 금지.
- 헤드라인, 오퍼, 샘플 화면을 먼저 개선.

### 경우 B — 모달 오픈은 높고 폼 완료가 낮음

- 2단계 폼, 필드 축소, 신뢰 문구 실험.

### 경우 C — 제출은 많지만 적합 리드가 낮음

- 타깃 문구와 폼 qualification 강화.
- 역할별 유입 분리.

### 경우 D — 적합 리드와 상담이 충분함

- V5 자가진단과 리포트 자동화 착수.

### 경우 E — 상담은 있으나 파일럿 관심이 낮음

- 제품 기능보다 유료 오퍼, 범위, 결과물, 가격 명목을 재검증.

## 14. 다음 버전에서 피할 실패

- 설문 결과 없이 전체 SaaS를 먼저 개발
- 모든 타깃에 다른 기능을 한 번에 제공
- 점수를 과학적 사실처럼 제시
- 무료 체크리스트 사용자와 구매 의향자를 동일하게 취급
- 가짜 대시보드 수치를 실제 성과처럼 사용
- 사용자의 메모나 이메일을 분석 도구로 전송
- 테스트 없이 새로운 CTA를 동시에 여러 곳에서 변경
