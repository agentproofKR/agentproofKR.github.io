# AgentProof 랜딩페이지 검수 과정 문서

- 문서 버전: 1.0
- 대상: V4.1 Production
- 목적: 구현 결과를 주관적 인상만이 아니라 기능, 시각, 데이터, 접근성, 보안 기준으로 검수한다.

---

## 1. 검수 원칙

- “화면이 비슷해 보인다”만으로 승인하지 않는다.
- 주요 전환인 폼 저장이 실제로 작동해야 한다.
- 성공 메시지는 서버 저장 성공 뒤에만 표시한다.
- 데스크톱과 모바일을 각각 독립적으로 검수한다.
- 샘플 수치와 실제 데이터의 구분을 확인한다.
- 개인정보가 분석·로그·오류 메시지에 노출되지 않는지 확인한다.

---

## 2. 검수 게이트

| Gate | 이름 | 통과 조건 |
|---|---|---|
| G0 | 요구사항 동결 | 문구, 섹션, 폼 필드, 이벤트 이름 확정 |
| G1 | 개발자 자체 검수 | lint/typecheck/unit/build 통과 |
| G2 | 기능 검수 | CTA, 모달, 폼, API, UTM 정상 |
| G3 | 시각·반응형 검수 | 참조 이미지 위계 유지, 주요 화면 오류 없음 |
| G4 | 접근성·성능·보안 | release blocker 없음 |
| G5 | Staging 승인 | 전체 시나리오와 증거 패키지 확인 |
| G6 | Production smoke | 실제 도메인에서 제출·분석·DB 확인 |

G0~G5 통과 전 production 배포를 승인하지 않는다.

## 4. 테스트 데이터

### 정상 데이터

```text
역할: 대표·임원·팀장
현재 단계: 회사 차원 도입 검토 중
걱정: 어떤 업무부터 도입할지 모르겠음
회사/팀명: QA 테스트 팀
이메일: qa+agentproof@example.com
상황 설명: 보고서 작성과 고객응대 업무부터 파일럿을 검토 중입니다.
```

### 경계 데이터

- 회사명 2자, 80자
- 이메일 254자에 가까운 정상값
- 메모 1,000자
- 한글, 영문, 숫자, 일반 구두점

### 비정상 데이터

- 빈 필수값
- 잘못된 이메일
- 허용 목록에 없는 select 값 직접 전송
- 1,001자 메모
- HTML/스크립트 문자열
- honeypot 값 존재
- 짧은 시간 반복 제출

실제 고객 이메일이나 기밀정보는 검수 데이터로 사용하지 않는다.

## 6. 분석 이벤트 검수

디버그 모드 또는 dataLayer inspector로 확인한다.

| ID | 이벤트 | 확인 항목 |
|---|---|---|
| AN-001 | page_view | variant와 UTM 존재 |

Release blocker:

- 이메일, 회사명, memo가 분석 payload에 포함됨.
- 성공 이벤트가 서버 저장 전 발생함.

## 7. 시각 검수

### 참조 이미지

- `reference/agentproof_v4_target_desktop.png`
- `reference/agentproof_v4_target_mobile.png`

### 비교 항목

- 헤더 높이와 좌우 배치
- 히어로의 줄바꿈, 크기, 여백
- CTA 우선순위
- 앱 화면의 비율과 다크/라이트 대비
- 역할 3개 카드의 정렬
- 판단 흐름의 연결감
- 다크 섹션의 시작 위치와 시각적 대비
- 모바일에서 가로 넘침 여부

### 허용 범위

- 시스템 폰트 차이로 인한 글자 폭 소폭 차이
- 브라우저 렌더러에 따른 1~2px 차이
- 세로 길이의 소폭 변화

### 허용하지 않는 차이

- H1이 다른 의미의 줄로 끊김
- CTA가 첫 화면 아래로 밀림
- 대시보드 글자가 읽을 수 없을 정도로 축소됨
- 모바일에서 카드 또는 모달이 화면 밖으로 나감
- 다크 섹션이 밝은 섹션과 구분되지 않음

## 14. 자동 테스트 최소 세트

### 단위 테스트

- leadSchema 정상·비정상 입력
- UTM 파싱과 보존
- analytics payload PII 차단
- repository 오류 처리

### E2E

1. 히어로 CTA → 모달 → 정상 제출 → 성공
2. 빈 폼 → 오류 → 수정 → 성공
3. UTM URL → 제출 → API payload 확인
4. API 500 mocking → 실패 메시지
5. 키보드로 모달 열기·작성·닫기
6. 모바일 뷰포트 정상 제출
7. 개인정보처리방침 이동

### 시각 회귀

- desktop 1440
- mobile 390
- modal desktop
- modal mobile

## 16. Production smoke test

배포 직후 다음 순서로 15분 이내 확인한다.

1. 실제 도메인 HTTPS 접속.
2. 메타데이터와 OG 확인.
3. 히어로·앵커·모달 확인.
4. UTM 테스트 URL로 정상 폼 제출.
5. DB 저장 확인.
6. 운영자 알림 확인.
7. success 이벤트 확인.
8. 테스트 리드를 삭제하거나 `test` 상태로 변경.
9. 모바일 실제 기기에서 폼 제출 없이 모달 동작 확인.

이 중 4~7이 실패하면 즉시 롤백 또는 외부 폼 fallback을 적용한다.
