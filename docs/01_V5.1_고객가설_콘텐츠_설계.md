# AgentProof V5.1 고객가설형 랜딩페이지 콘텐츠 설계

## 1. 이번 수정의 결론

V5의 시각 구조와 MVP 대시보드는 유지하되, 랜딩페이지의 역할을 **완성된 SaaS 판매 페이지**에서 **문제제기 + 고객가설 검증 페이지**로 바로잡는다.

AgentProof의 장기 비전은 업무용 AI/AI Agent를 만들고, 배포 전 검증하고, 운영 중 모니터링하며, 승인과 기록까지 남기는 신뢰 플랫폼이다. 초기 MVP는 사내 문서·업무규정 검색 Agent와 검증 대시보드다. 그러나 현재 단계에서 먼저 확인해야 할 것은 기능의 완성도가 아니라 다음 세 가지다.

1. 누가 가장 강한 문제를 느끼는가.
2. 각 역할이 원하는 첫 결과물은 무엇인가.
3. 누가 체크리스트를 넘어 샘플 리포트·파일럿 상담까지 요청하는가.

따라서 한 화면을 세 팀이 공유한다는 서사가 아니라, **세 역할마다 다른 문제와 첫 서비스 가설을 제시하고 반응을 따로 측정**한다.

---

## 2. 프로젝트 목표와 랜딩페이지 목표 분리

### 프로젝트 목표

업무용 AI를 실제 조직에 적용할 때 필요한 답변 근거, 정확성, 정책, 권한, 승인, 운영 기록을 검증할 수 있는 서비스와 제품을 만든다.

### 현재 MVP

사내 문서·업무규정 검색 Agent + 근거·오답·정책·권한 검증 대시보드.

### 이번 랜딩페이지 목표

제품을 완성품처럼 판매하지 않는다. 세 고객군의 문제 반응과 후속 행동을 측정해 초기 고객과 첫 유료 서비스를 결정한다.

- 문제 카드 선택
- 역할별 진단 시작
- 체크리스트 요청
- 샘플 리포트 요청
- 파일럿 상담 요청

---

## 3. 고객가설

| 고객 | 핵심 문제 가설 | 필요한 첫 서비스 가설 | 강한 검증 신호 |
|---|---|---|---|
| 실무자 | AI 답을 어디까지 믿고, 회사 자료를 어디까지 넣어도 되는지 모른다 | 근거 확인 + 업무 AI 사용 가이드 | 역할별 체크리스트 요청, 실제 업무 사례 입력 |
| 대표·도입 담당자 | 어떤 업무부터 시작해야 효과와 위험을 설명할 수 있는지 모른다 | AI 도입 우선순위 + 준비도 진단 | 샘플 리포트 요청, 파일럿 상담 |
| 보안·정책 담당자 | 개인정보·권한·승인·로그 기준이 없거나 기존 규정에 AI가 반영되지 않았다 | AI 사용정책 + 권한·기록 점검 | 정책 샘플 요청, 보안 파일럿 상담 |

세 고객군을 같은 구매자로 가정하지 않는다. 실무자는 문제 근거와 내부 확산의 출발점일 수 있고, 대표·도입 담당자 또는 보안·정책 담당자가 실제 예산권자가 될 수 있다. 랜딩페이지 데이터로 이 관계를 확인한다.

## 4. 실제 서비스 페이지에서 가져온 원칙

- Linear, Credal, Glean은 첫 화면에서 한 가지 가치만 말하고, 헤드라인 다음에 한 문장과 한 개의 주 CTA를 둔다.
- Notion은 추상적인 AI 효익보다 실제 질문과 작업 예시를 보여주는 방향으로 카피를 바꿨다.
- Notion, Credal, Glean은 제품 화면을 설명문보다 먼저 또는 매우 이른 구간에 배치한다.
- OneTrust, IBM watsonx.governance는 거버넌스를 역할 이름보다 구체적인 행위—검토, 평가, 통제, 모니터링—로 설명한다.

AgentProof 적용 원칙:

1. 히어로는 하나의 질문만 남긴다.
2. 서브카피는 한 문장으로 제한한다.
3. 제품 화면은 유지하되 제품 콘셉트임을 표시한다.
4. 역할별 영역은 설명이 아니라 `문제 → 필요한 서비스 → 선택`으로 구성한다.
5. 마지막 CTA는 판매 요청이 아니라 고객가설 검증 행동으로 연결한다.

## 5. 글자 수 기준

| 요소 | 기준 |
|---|---:|
| H1 | 한 가지 주장, 데스크톱 2줄 이내 |
| 히어로 설명 | 1문장, 약 45~65자 |
| 섹션 제목 | 약 12~24자 |
| 역할별 문제 | 질문 1개, 약 15~24자 |
| 역할별 서비스 | 명사형 2개 이하 |
| 단계 설명 | 1문장, 약 25자 이하 |
| CTA | 8~12자 |

V5 대비 주요 마케팅 문구는 약 37% 축소했다. 대시보드 안의 샘플 UI 텍스트는 제품 이해에 필요하므로 유지했다.

## 6. 확정 카피

### 히어로

**업무 AI, 어디까지 맡겨도 될까요?**

실무자의 사용 기준, 도입 담당자의 우선순위, 보안 담당자의 통제 기준을 진단합니다.

- 주 CTA: `내 과제 3분 진단`
- 보조 CTA: `MVP 화면 보기`

### 고객가설 섹션

**어디에서 가장 막히시나요?**

역할마다 필요한 첫 서비스가 다릅니다.

- 실무자: “AI 답을 믿고 써도 될까요?” → 근거 확인 · 업무 사용 가이드
- 대표·도입 담당자: “어떤 업무부터 시작해야 할까요?” → 도입 우선순위 · 준비도 진단
- 보안·정책 담당자: “무엇을 막고 기록해야 할까요?” → 사용정책 · 권한·기록 점검

### MVP 섹션

**첫 파일럿은 문서·규정 검색부터.**

답변, 근거, 정책 위험을 실제 질문으로 검증합니다.

- 업무 1개 선택
- 검증
- 결과

### 마지막 CTA

**지금 가장 필요한 서비스를 알려주세요.**

응답을 바탕으로 첫 고객과 기능 우선순위를 정합니다.

## 7. 폼과 측정

폼은 다음 항목만 받는다.

1. 역할
2. 현재 AI 도입 단계
3. 가장 가까운 문제
4. 원하는 다음 단계: 체크리스트 / 샘플 리포트 / 파일럿 상담
5. 업무 이메일
6. 적용 업무(선택)

필수 이벤트:

- `role_problem_click`
- `diagnosis_start`
- `diagnosis_submit`
- `followup_checklist`
- `followup_sample_report`
- `followup_pilot`

역할 카드 클릭 시 역할을 미리 선택하고 모달을 연다. 조회수보다 역할별 샘플 리포트 및 파일럿 상담 비율을 우선 지표로 본다.

---

## 8. 구현 기준

- `AgentProof` 브랜드와 현재 MVP 대시보드 이미지를 유지한다.
- 대시보드에는 `SAMPLE DATA` 또는 샘플 데이터 안내를 유지한다.
- 고객 로고, 후기, 인증, 성과 수치를 임의로 추가하지 않는다.
- 모바일 390px에서 가로 스크롤이 없어야 한다.
- 세 역할 카드는 각각 다른 placement와 role 값을 전송해야 한다.
- 현재 화면은 Private beta 고객검증용임을 폼과 푸터에서 명확히 알린다.
