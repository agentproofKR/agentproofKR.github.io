# AgentProof 랜딩페이지 개발계획문서

- 문서 버전: 1.0
- 대상 릴리스: V4.1 Production
- 선행 문서: `01_랜딩페이지_설계문서.md`
- 목표: Codex가 저장소 생성부터 배포 검증까지 순차 실행할 수 있는 구현 계획

---

## 1. 개발 원칙

1. **짧고 정확하게 구현한다.** 랜딩페이지에 불필요한 프레임워크와 기능을 추가하지 않는다.
2. **문구를 임의 수정하지 않는다.** 카피 변경은 별도 이슈로 분리한다.
3. **샘플과 실제 기능을 구분한다.** 진단 대시보드는 샘플이고, 리드 제출은 실제 기능이다.
4. **서버 검증을 신뢰 기준으로 둔다.** 클라이언트 검증만으로 입력을 저장하지 않는다.
5. **접근성·분석·개인정보를 처음부터 포함한다.** 출시 직전에 붙이는 부가 작업으로 취급하지 않는다.
6. **정적 자산을 가볍게 유지한다.** 불필요한 영상, 대형 라이브러리, 외부 폰트 파일을 피한다.

## 2. 권장 기술 스택

정확한 패키지 버전은 개발 착수 시점의 안정 버전을 사용하고 lockfile로 고정한다.

| 영역 | 선택 |
|---|---|
| 프레임워크 | Next.js App Router + React + TypeScript strict |
| 패키지 매니저 | pnpm |
| 스타일 | CSS Modules + `app/globals.css` 디자인 토큰 |
| 폼 | React Hook Form + Zod |
| API | Next.js Route Handler `/api/leads` |
| 데이터 저장 | Supabase Postgres 기본 어댑터 |
| 알림 | Resend 이메일 알림 선택 기능 |
| 분석 | Google Tag Manager/GA4 또는 동일 이벤트 인터페이스 |
| E2E | Playwright |
| 단위 테스트 | Vitest 또는 프로젝트 기본 테스트 러너 |
| 배포 | Vercel 권장 |
| 코드 품질 | ESLint + Prettier + TypeScript noEmit |

### 선택 이유

- 한 페이지이지만 폼 서버 처리, SEO, 추후 진단 결과 페이지 확장이 필요하다.
- CSS Modules는 현재 시안의 정교한 반응형 레이아웃을 재현하면서 런타임 의존성을 줄인다.
- 데이터 저장소는 인터페이스로 분리해 Supabase 이외 서비스로 교체 가능하게 한다.

```bash

LEAD_NOTIFICATION_TO=
LEAD_NOTIFICATION_FROM=
```

규칙:

- `SUPABASE_SERVICE_ROLE_KEY`, `RESEND_API_KEY`, `TURNSTILE_SECRET_KEY`는 서버 전용이다.
- 선택 기능이 비어 있어도 로컬 개발은 가능해야 한다.
- 필수 환경 변수는 `lib/env.ts`에서 시작 시 검증한다.
- 개발 환경에서는 저장 어댑터를 명시적으로 `mock`으로 설정할 수 있게 한다.

## 5. 데이터 모델

### 5.1 leads 테이블

```sql
create table if not exists public.leads (
  id uuid primary key default gen_random_uuid(),
  created_at timestamptz not null default now(),
  role text not null,
  stage text not null,
  concern text not null,
  company text not null,
  email text not null,
  memo text,
  consent_version text not null,
  consented_at timestamptz not null,
  utm_source text,
  utm_medium text,
  utm_campaign text,
  utm_content text,
  landing_variant text not null default 'v4.1',
  referrer text,
  status text not null default 'new',
  source_page text not null default '/',
  metadata jsonb not null default '{}'::jsonb
);

create index if not exists leads_created_at_idx on public.leads (created_at desc);
create index if not exists leads_status_idx on public.leads (status);
create index if not exists leads_email_idx on public.leads (lower(email));
```

### 5.2 저장 금지 데이터

- 원문 IP 주소
- 실제 내부 문서
- 비밀번호 또는 API 키
- 건강·주민등록·금융 등 민감정보
- 자유 입력 메모의 내용을 분석 이벤트로 복제한 값

### 5.3 데이터 정규화

- 이메일은 소문자로 저장한다.
- 회사명·메모는 앞뒤 공백을 제거한다.
- select 값은 서버 허용 목록으로 검증한다.
- memo는 HTML을 허용하지 않고 일반 문자열로 저장한다.

## 6. API 계약

## 6.1 Endpoint

`POST /api/leads`

### Request

```json
{
  "role": "대표·임원·팀장",
  "stage": "회사 차원 도입 검토 중",
  "concern": "어떤 업무부터 도입할지 모르겠음",
  "company": "AgentProof Lab",
  "email": "name@example.com",
  "memo": "부서 파일럿 전 기준이 필요합니다.",
  "consent": true,
  "consentVersion": "2026-06-18",
  "utm": {
    "source": "linkedin",
    "medium": "social",
    "campaign": "launch",
    "content": "founder-post-01"
  },
  "landingVariant": "v4.1",
  "honeypot": ""
}
```

### Success 201

```json
{
  "ok": true,
  "leadId": "uuid",
  "message": "신청이 완료되었습니다."
}
```

### Validation error 400

```json
{
  "ok": false,
  "code": "VALIDATION_ERROR",
  "fieldErrors": {
    "email": "업무 이메일 형식을 확인해주세요."
  }
}
```

### Rate limit 429

```json
{
  "ok": false,
  "code": "RATE_LIMITED",
  "message": "잠시 후 다시 시도해주세요."
}
```

### Server error 500

사용자에게 내부 오류 상세를 노출하지 않는다.

## 6.2 처리 순서

1. JSON 크기 제한 확인.
2. honeypot 검사.
3. Zod 서버 검증.
4. rate limit 확인.
5. 텍스트 정규화.
6. repository 저장.
7. 선택적으로 운영자 이메일 알림.
8. 201 응답.
9. 로그에는 lead ID와 오류 코드만 기록하고 이메일·메모는 남기지 않는다.

## 7. 분석 구현

`lib/analytics.ts`에 아래 인터페이스를 둔다.

```ts
export type AnalyticsEvent =
  | 'diagnostic_preview_click'
  | 'lead_modal_open'
  | 'lead_form_start'
  | 'lead_form_submit_attempt'
  | 'lead_form_validation_error'
  | 'lead_form_success'
  | 'lead_form_server_error'
  | 'nav_anchor_click';

export function trackEvent(
  event: AnalyticsEvent,
  payload?: Record<string, string | number | boolean | string[]>
): void;
```

규칙:

- GTM이 없으면 함수는 오류 없이 no-op.
- 개인정보 필드는 payload에 허용하지 않는다.
- 모든 CTA에 `placement`를 넣는다: `header`, `hero`, `process`.
- UTM은 URL에서 읽고 sessionStorage에 보존한다.
- 페이지 내 이동 후에도 최초 유입 UTM을 유지한다.

## 8. 컴포넌트 구현 기준

### Header

- sticky.
- 모바일에서는 내비게이션 링크 숨김.
- 현재 페이지 내 anchor에만 사용.

### HeroSection

- H1은 페이지당 하나.
- CTA 두 개.
- 텍스트 너비와 줄바꿈은 설계문서 기준.

### DiagnosticPreview

- 실제 HTML/CSS로 구성.
- `SAMPLE DATA` 고지 고정.
- 장식 아이콘은 inline SVG 또는 CSS.
- 의미 없는 외부 이미지 의존 금지.

### RoleValueSection

- 3개 역할 카드.
- 모바일 1열.
- DecisionFlow를 별도 하위 컴포넌트로 사용.

### ProcessSection

- 다크 테마.
- 3단계.
- 최종 CTA.

### Modal

- `role="dialog"`, `aria-modal="true"`.
- 포커스 트랩과 포커스 복귀.
- 스크롤 잠금.
- reduced motion 지원.

### LeadForm

- React Hook Form + Zod.
- 클라이언트와 서버 스키마를 공유.
- 첫 입력 시 한 번만 `lead_form_start` 전송.
- 서버 성공 전에는 성공 상태를 표시하지 않는다.
- 제출 중 재클릭 방지.

## 10. 구현 순서

### Phase 0 — 준비

- 참조 자료 확인.
- 문구를 상수 또는 콘텐츠 객체로 분리.
- `.env.example` 작성.
- CI에서 lint, typecheck, test, build 실행.

### Phase 1 — 정적 UI

- Header, Hero, DiagnosticPreview.
- 역할별 가치, 판단 흐름, Process, Footer.
- 참조 이미지와 1차 시각 비교.

### Phase 2 — 폼 상호작용

- 접근 가능한 Modal.
- 폼 검증과 상태.
- UTM 수집.
- 이벤트 추적 인터페이스.

### Phase 3 — 실제 저장

- `/api/leads`.
- Supabase repository.
- 운영자 알림.
- rate limit과 spam 방지.

### Phase 4 — 품질

- 반응형.
- 접근성.
- 성능.
- SEO.
- 테스트 자동화.

### Phase 5 — 출시

- staging 검수.
- 환경 변수 등록.
- production 배포.
- 실제 폼 제출·DB·알림·분석 smoke test.

## 14. 배포 체크

### Staging

- robots noindex 권장.
- 테스트용 DB 또는 status 구분.
- 실제 이메일을 운영자에게 보내지 않도록 설정 가능.

### Production

- canonical과 `NEXT_PUBLIC_SITE_URL` 확인.
- 운영 DB.
- GTM production ID.
- 개인정보처리방침 공개.
- 도메인 HTTPS.
- 폼 실제 제출 1건 후 삭제 또는 테스트 상태 표시.

### 롤백

- 직전 성공 배포를 유지한다.
- DB 마이그레이션은 additive 방식 우선.
- 폼 장애 시 CTA를 임시 외부 폼으로 연결할 수 있는 환경 변수 fallback을 준비한다.

## 15. Definition of Done

티켓 전체 완료 조건:

- 설계문서의 확정 문구와 구조가 구현됨.
- 320px 이상에서 가로 스크롤 없음.
- 폼이 서버와 DB에 실제 저장됨.
- 실패 시 거짓 성공 메시지를 표시하지 않음.
- UTM과 landing variant가 저장됨.
- 분석에 개인정보가 전송되지 않음.
- 키보드만으로 모든 기능 사용 가능.
- 핵심 E2E 테스트 통과.
- QA 문서의 P0/P1 release blocker 0건.
- production smoke test 증거가 남음.
